# Changelog

## [Unreleased]

### Added

- **First-Time Onboarding Wizard** ‚Äî Implemented `/onboarding` page with 3-step wizard for new users. Step 1: Create new household or join existing one. Step 2: Add family members (spouse, children, etc.) with relationship and age - skippable. Step 3: Set communication preference (Email/SMS/Both/None). Added `onboardingCompletedAt` field to User model. Created `WizardStep` component for reusable wizard UI. Created `HelpButton` component with context-aware help panel (persistent "?" button). Created API routes: `/api/onboarding/household` for household creation/joining, `/api/onboarding/dependent` for adding family members, `/api/onboarding/complete` for marking onboarding done and setting communication preference. Added `completeOnboarding` mutation to user router. Ticket 21 Done.
- **Audit Middleware for Admin Procedures** ‚Äî Fixed TypeScript typing issues that prevented adding audit middleware to the adminProcedure chain. Added `AuthedCtx` interface with non-null session type, updated `isAuthenticated` and `isAdmin` middleware with type assertions to preserve session narrowing. Added `auditLog` middleware that writes `AdminAuditLog` entries for all admin mutations. Created `auditedAdminProcedure` that chains all three middleware. Migrated all admin routers (admin, communication, event, invitation, potluck, rsvp) to use `auditedAdminProcedure`. Every admin mutation now automatically produces an audit log entry with action path. Tickets 18 and 05 now Done.
- **Dietary Label Filtering** ‚Äî Implemented dietary label filtering on event detail page and admin dashboard. Created `DietaryLabelChip` component with standard label set (vegetarian, vegan, gluten_free, contains_nuts, dairy_free), `DietaryFilter` component for selecting dietary needs, and `DietaryAttendeeFilter` component for displaying filtered attendees. Event detail page (`/events/[id]`) now shows dietary aggregation summary and filter chips to show only attendees with specific dietary restrictions. Admin dashboard (`/admin/dashboard`) now displays dietary restriction counts per event with color-coded chips. Parses dietary notes for common keywords to extract labels. Ticket 22 Done.
- **Events Calendar View** ‚Äî Implemented `/events/calendar` page with month grid calendar showing all PUBLISHED events. Created `Calendar` component with month navigation (previous/next/today), today highlighting with green ring, event chips color-coded by status (PUBLISHED=green, DRAFT=amber, CLOSED=gray, CANCELLED=red with strikethrough), "show more" overflow handling when >3 events per day, and mobile-friendly responsive design with min-height 24/32 cells. Created `CalendarEventChip` component with link to event detail. Calendar legend shows status color meanings. Navigation between events list and calendar view. Ticket 23 Done.
- **Component Hook Migration Complete** ‚Äî Migrated all four components from raw fetch to tRPC hooks: RSVPForm now uses `useRsvpMutation` hook (confirm/decline mutations with automatic query invalidation), ProfileClient now uses `useUserProfileMutation` and `useDependentMutations` hooks, PhotoReactionButton now uses `usePhotoReactionMutation` hook (add/remove reactions). Created new hooks: `useRsvp.ts` (useRsvpMutation), `useUser.ts` (useUserProfileMutation), `usePhoto.ts` (usePhotoReactionMutation). All hooks use React Query semantics with onSuccess query invalidation. Ticket 16 now Done.
- **PotluckSignupForm migrated to use tRPC hooks** ‚Äî PotluckSignupForm now uses `usePotluckSignupMutation` hook instead of raw fetch, with proper loading/error states via React Query semantics. Also fixed `potluck.signup` tRPC mutation to properly update `currentSignups` counter with serializable transaction for LIMITED slots (matching API route behavior). The signup mutation now handles race conditions atomically for LIMITED slots. Ticket 16 Partial complete.

### Fixed

- **Potluck slot race condition** ‚Äî Fixed race condition in `src/app/api/potluck-signup/route.ts` by wrapping signup + counter increment in `prisma.$transaction` with `isolationLevel: Serializable`. The count check, signup create/update, and counter increment now happen atomically. Returns 409 Conflict when slot is full instead of allowing over-signup.
- **Auto-release potluck slots on RSVP decline** ‚Äî When an RSVP is declined, potluck signups are now automatically released. The implementation in `src/app/api/rsvp/route.ts` decrements `PotluckSlot.currentSignups` by the signup's servings count, deletes all `PotluckSignup` rows tied to the RSVP, updates the RSVP to DECLINED status with headcount 0, and writes an audit log entry with `POTLUCK_SLOT_RELEASE` action ‚Äî all within a single transaction for atomicity.
- **Remove duplicate NextAuth handler** ‚Äî Removed duplicate `GET`/`POST` handler exports from `src/lib/auth.ts`. The active handler lives in `src/app/api/auth/[...nextauth]/route.ts` which imports `authOptions` from lib. The lib now only exports `authOptions` and `getServerSession`. Auth test file updated to test current exports instead of removed handlers. TypeScript errors fixed in `src/lib/__tests__/auth.test.ts`.
